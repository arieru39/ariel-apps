## 序章｜なぜ "設計思想" を先に渡すのか

`book-to-app` の11テーマは、Claudianが自動生成できます。
でも、生成された **ルーブリックの4段階を、自分の現場の1行動として書き直せるかどうか** は、あなたの中に "設計思想" があるかどうかで決まります。

設計思想を持たないままアプリを改造すると：
- ルーブリックが「できる／できない」の**評価表**に退化する
- 観点クイズに **正解** を仕込みたくなる
- 動画が **要約** になる
- アプリが「触る道具」ではなく「見せる作品」になる

一方、設計思想を持てば：
- ルーブリックは **旅の地図** として現場と繋がる
- 観点クイズは **位相を映す鏡** になる
- 動画は **段階遷移の触媒** になる
- アプリは **対話の入り口** として残り続ける

この違いを生むのは、**認知科学の系譜** です。Ericsson が明らかにした「熟達は才能ではない」という発見、Krashen の「届く言葉」の理論、Kahneman の Fast/Slow、Chi et al. の「初心者と熟達者は同じ問題を違う目で見ている」という実証。**これらを1つも読まずにアプリを触ることもできます。でも、1冊読むだけで、あなたの改造は別物になります**。

## 第1部｜4段階ルーブリックは熟達研究から立ち上がっている

### 1.1 Ericsson の発見 ─ 「才能」の物語を捨てる

**Anders Ericsson**（1947-2020、フロリダ州立大学）は、生涯を「専門的能力（Expertise）とは何か」の研究に捧げました。ヴァイオリニスト、チェス・プレーヤー、外科医、テニス選手、記憶術の達人 ── 領域を横断して「トップに立つ人は何が違うのか」を実証的に調べ続けた。

彼の結論は、当時の常識を覆すものでした。

> **「熟達とは、才能ではなく、"目的を持った練習（Purposeful Practice）" と、それを支える "心的表象（Mental Representation）" の質で決まる」**

彼の代表作『Peak: Secrets from the New Science of Expertise』（邦訳『超一流になるのは才能か努力か?』、2016）は、「1万時間の法則」（Gladwell）が **単純化しすぎている** ことを指摘しました。1万時間かければ誰でも達人になるのではない。**「同じことを1万時間繰り返す」のと「意図的に少し難しい課題に挑戦し続ける1万時間」は違う** のです。

### 1.2 Chi, Feltovich & Glaser 1981 ─ 「見え方」が違う

Ericsson の理論を実証的に補強した古典が、Michelene Chi, Paul Feltovich, Robert Glaser の1981年の研究です。

彼らは「物理の問題」を初心者（学部生）と熟達者（PhD持ちの物理学者）に見せて、**問題を分類させる** 実験をしました。結果：

- **初心者**：問題文の表面（「斜面がある問題」「バネがある問題」）で分類した
- **熟達者**：物理原理（「エネルギー保存則が使える問題」「ニュートン第2法則が使える問題」）で分類した

つまり、**同じ問題を見ても、初心者と熟達者は "違うものを見ている"**。熟達者の頭の中には、原理を軸にした **心的表象（Mental Representation）** があり、それが認知そのものを組み替えている。

**これが `book-to-app` のルーブリック4段階の正体です**：
- L1（気づいていない）＝ 表面の症状で反応する初心者
- L2（概念理解）＝ 教科書のフレームワークを"当てはめる"レベル
- L3（実践）＝ 現場の固有性を読み取り、フレームの外側を掴める
- L4（選び分けられる）＝ 心的表象そのものから状況を構造化できる達人

### 1.3 心的表象（Mental Representation）とは何か

Ericsson の中心概念は「心的表象」です。これは日常語では:
- 医師が「レントゲンを見る目」
- チェス・プレーヤーが「駒の配置を "一手の意味" として見る目」
- ファシリテーターが「場の空気を "誰が誰の壁になっているか" として見る目」

抽象的に言えば、**"情報を意味のある単位（chunk）として構造化して保持する能力"**。この心的表象は生まれつきではなく、**意図的な練習を通じて構築される** ものです。

だからルーブリックL4は「頭がいい人」の記述ではなく、**"その領域の心的表象を持っている人"** の記述として書きます。

### 1.4 だから4段階は「評価」ではなく「地図」

`book-to-app` のルーブリックが**判定表**ではなく**旅の地図** である理由は、心的表象は「持っている／持っていない」の2分法ではなく、**構築の途中にある** 動的なものだからです。

L1のあなたと L4 のあなたは、**同じ人生の中で共存します**。ある領域では L4 の心的表象を持っていても、別の領域では L1 の反射で動いてしまう。L4は「到達したら終わり」ではなく、**毎日再更新される熟達の位相**（[[_rubric_コンサルタントコーチとしての熟達]] の言葉）です。

> **🌱 深めるための問い**：あなたが自分の本を選ぶ時、その本の主張のうち「自分がL4に居られる領域」と「自分がL1で誤作動している領域」は、それぞれどこか？

## 第2部｜観点クイズが判定色を出さない理由 ─ Comprehensible Input

### 2.1 Krashen の Input Hypothesis

**Stephen Krashen**（南カリフォルニア大学名誉教授）は第二言語習得研究の巨人です。彼の「Input Hypothesis」（1985）は、外国語学習だけでなく、**あらゆる学習の設計** に応用されています。

中心命題は驚くほどシンプル：

> **「学習者は、"少しだけ現在地の先"（i+1）にある、意味のわかる（Comprehensible）Input を受けることで習得する」**

- `i` は学習者の現在地
- `i+1` は「一段先」の入力
- `i+2` 以上は **理解できないので習得に繋がらない**
- `i` と同じレベルは **既知の反復で成長しない**

つまり、教育の質は **「相手の現在地を正確に読み、そこから一段先の入力を提供できるか」** で決まる。

### 2.2 「届く」ことと「わかる」ことの違い

Comprehensible Input のポイントは、**内容が正しい** ことではなく、**相手に届く** ことです。教科書的に正確でも、学習者の現在地から遠すぎる言葉は、"存在しないのと同じ" になる。

これが `book-to-app` の観点クイズが **判定色（正解／不正解の色分け）を出さない** 理由です。判定色は「あなたが正しいか間違っているか」を教える。でも Krashen によれば、学習を進めるのは判定ではなく、**"あなたの現在地から一段先" が見える瞬間** です。

だから観点クイズの選択肢はすべて **「選んでも正解にも不正解にもならない」**。代わりに、選んだ選択肢に応じて **stance（あなたの現在の視点）／phase（あなたの位相）／author_view（本の一段先の言葉）／slow_question（一段先の身体感覚）** が返る。これが Comprehensible Input の実装です。

### 2.3 author_view は「一段先の言語」

`perspective_quiz` の各選択肢に必ずある `author_view` は、**本文からの具体的な引用** です。抽象的な要約ではなく、"本を開けば戻れる一節" を返す。

なぜか？ Krashen の i+1 が実現するには、**あなたが受け取れる形の "一段先" が届く必要がある**。抽象要約は多くの場合 i+2 以上に飛ぶ（読者との現在地の距離が測れない）。**書籍の具体箇所は "著者が読者の現在地から書き起こした一段先"** であることが多い。だから引用が最も安全な i+1 の供給源になる。

### 2.4 slow_question は「一段先の身体」

`author_view` が「言語の一段先」なら、`slow_question` は「身体の一段先」です。

Krashen の理論はもともと言語に焦点を当てていましたが、**熟達の Input は言語だけではありません**。姿勢・呼吸・視線・間 ── これらも学習者の位相を動かす。だから `book-to-app` は各選択肢に **"あなたが今日一つ動ける重さの問い"** を添える。

**重要な設計原則**：`slow_question` は "二段先" ではなく **"一段先"** を狙う。二段先の問いは届かない。「その本に戻るとき、あなたは何を確かめに戻っているか？」は届くが、「あなたの心的表象を書き換えると全体が動く一点はどこか？」はL2の人には届かない。

> **🌱 深めるための問い**：あなたの観点クイズの1問目、`slow_question` は本当に "一段先" か？ 二段先に飛んでいないか？ 相手の身体で確かめる方法はあるか？

## 第3部｜Deliberate Practice の6要件をアプリに埋め込む

### 3.1 Ericsson の6要件

Ericsson は熟達研究から、**"目的を持った練習（Deliberate Practice）"** に共通する6要件を抽出しました。この6要件は「才能ある人が偶然やっていたこと」ではなく、**誰でも設計できる構造** です。

| # | 要件 | 意味 |
|---|---|---|
| 1 | **Comprehensible Input**（届く言葉） | 学習者の現在地から一段先の入力を提供する |
| 2 | **即時のフィードバック** | 正しかったか／逸れたかを、行動の直後に返す |
| 3 | **明確で具体的な目標** | 抽象な「上手くなる」ではなく「〜を判別できる」の形 |
| 4 | **完全な集中** | 気が散る状態では熟達回路は組まれない |
| 5 | **コンフォートゾーンの一段外** | 楽な領域では成長せず、苦しすぎる領域では逃げる |
| 6 | **専門家による設計** | 独学では気づけない盲点を、達人の設計が埋める |

### 3.2 各要件がアプリのどこに実装されているか

`book-to-app`（および同じ設計思想の tamakatsu / KPI / NVC / talkable-school など）は、この6要件が **同時に満たされる場** として設計されています。

| 要件 | アプリでの実装 |
|---|---|
| Comprehensible Input | `author_view` の本文引用 ＋ `slow_question` の一段先の問い |
| 即時のフィードバック | 選択肢を選んだ瞬間に stance/phase/author_view/slow_question の5層が展開 |
| 明確な目標 | 各設問頭の `learning_objective`（「〜を掴む」の形で1行） |
| 完全な集中 | 1問1画面・判定色なし・不要情報を排除したUI |
| コンフォートゾーンの一段外 | `slow_question` が現在地を一段だけ押し上げる |
| 専門家の設計 | `author_view` が **書籍（専門家）の視座** そのもの |

**このマッピングを覚えたら、あなたは自分の観点クイズを設計できる**。ある選択肢が「なんとなく違和感がある」時、**"6要件のどれが弱いか"** で診断できるようになります。

### 3.3 コンフォートゾーンの「一段外」の設計

6要件で最も設計が難しいのが **"一段外（learning zone）"** の見極めです。

- **Comfort Zone**（楽）→ 反復しても成長しない
- **Learning Zone**（一段外）→ 意図的な負荷で成長する
- **Panic Zone**（二段以上先）→ 恐れが先に立って学習が止まる

`slow_question` の設計原則が **"一段先"** に留まる理由はここにあります。二段先の問いは Panic Zone に押し出す。**あなたの読者を Learning Zone に置き続ける** ためには、**あなた自身がその読者の現在地を身体で分かっている** 必要がある。だから「自分の現場のL1の人が今日やっている具体行動を1つ書く」がルーブリック設計の入り口になる。

> **🌱 深めるための問い**：あなたが最後に "一段外" にいた経験は何か？ そのとき、誰が／何が、あなたを一段外に置き続けたか？

## 第4部｜Fast と Slow の位相 ─ Kahneman の二重過程

### 4.1 System1 と System2

**Daniel Kahneman**（2002年ノーベル経済学賞）の『Thinking, Fast and Slow』（邦訳『ファスト＆スロー』、2011）は、人間の認知に2つのシステムがあると論じます。

- **System1（Fast）**：自動・高速・無意識・反射的。日常のほとんどはこれで動いている
- **System2（Slow）**：意識・遅い・エネルギー消費・熟慮

Fast は効率的ですが、**"認知バイアス" の温床** でもあります。一方 Slow は正確ですが、遅く・疲れる。熟達研究の重要な発見は：

> **熟達者は Slow で意識的に検証した経験を反復することで、その判断を Fast に "沈殿" させている**

医師が「レントゲンを一目見て診断できる」のは、過去何千回も Slow で検討した経験が Fast に沈殿しているから。チェス・グランドマスターの一瞥は、無数の Slow の解析の結晶です。

### 4.2 「沈殿」というメタファ

`book-to-app` の `engine_note` に必ず入る一節：

> **「Slow で意図的に検証・失敗・修正した経験が、Fast に沈殿する」**

これがアプリの中核原理です。学習アプリは Fast の反射を鍛える装置ではありません。**Slow の意識的な検証を反復する装置** です。反復した結果、**あなたの Fast の反射が高解像度になる（educated guess の域に達する）** のがL4の状態。

だから：
- **L1**：Fast が誤作動している（Slow が起動していない）
- **L2**：Slow で意識すれば使えるが、Fast にはまだ沈殿していない
- **L3**：Slow の反復で Fast に沈殿し始めた
- **L4**：Fast が高解像度（熟達直観）

これが rubric_labels の `fast_slow` フィールドの正体です。ルーブリックは "段階" だけでなく "認知の位相" を記述している。

### 4.3 4択の位相配分 ─ なぜ Fast:Slow:身体 = 1:2:1 か

`perspective_quiz` の4択には、**位相の目安の配分** があります：

- ⚡ **Fast の反射**：1つ（日常の即断・脊髄反射）
- 🐢 **Slow の再構築**：2つ（内省的な組み立て直し）
- 🌊 **身体の感覚**：1つ（ソマティックな受信）

なぜ Slow が2つで、Fast と身体が1つずつか？ 学習アプリの目的は **Slow の反復** です。だから **Slow の視点を2種類提示して "自分に届く方の Slow" を選ばせる** 設計にする。Fast は「日常の自分」の反射として1つあれば十分（自覚のため）。身体は言語化前の受信として1つ添える（言葉になっていない知恵を掬うため）。

**ここに一つ深い設計思想が入っています**：`book-to-app` は Kahneman の二項対立（Fast/Slow）に、**"身体"（Somatic）の第3項** を追加しています。Kahneman は認知科学者なので身体を主題化しませんでしたが、**熟達現場は身体で動く**（Gladwell『Blink』、Damasio、Feldenkrais の系譜）。だから3項の位相で読者の現在地を映す。

> **🌱 深めるための問い**：あなたの4択の中で、"身体の感覚" の選択肢は本当に身体か？ Slow の変装ではないか？

### 4.4 Klein × Kahneman 2009 ─ 直観の"良い場面"と"悪い場面"

もう一つ重要な参照：**Gary Klein × Kahneman 2009** の対話論文「Conditions for Intuitive Expertise」。二人はもともと "熟達直観" について立場が違いました（Klein は肯定、Kahneman は懐疑）。しかし対話の末に **合意した条件** がある：

> **直観が信頼できるのは：**
> 1. 環境に **規則性** がある領域（チェス、消防、医療）
> 2. その規則性が **フィードバック** で学習可能な領域

逆に言うと、**規則性のない領域や、フィードバックが遅すぎる領域** では、どれだけ経験を積んでも熟達直観は育たない（政治予測、株式市場、長期の組織文化変革の一部）。

これがアプリ設計に効く：**あなたの本の領域は、直観が育つ領域か？** 育たない領域なら、L4の記述は "直観の高解像度" ではなく、"謙虚に System2 を起動し続ける在り方" として書くべきです。

## 第5部｜Being と Doing ─ なぜ第3層に "在り方" を置くのか

### 5.1 Erhard & Jensen の Being a Leader

`book-to-app` の第3層（deploy / touch_together / gc_loop）や、`_rubric_コンサルタントコーチとしての熟達` の第3領域（🌱 Being）には、**"在り方"** という概念が置かれています。

これは **Werner Erhard × Michael Jensen** の「Being a Leader」コース（ハーバード等で教えられた）に由来する区別です。

- **Doing**（何をやったか）：観察可能・記述可能・複製可能な行動
- **Being**（何者としてそこに立つか）：観察不可能・記述されるより "顕れる" もの

Erhard & Jensen の中心命題：

> **「効果的なリーダーは、Doing ではなく Being から立ち上がる」**

同じ行動（Doing）でも、**"どんな Being から出た行動か"** で相手への届き方が違う。「頑張れ」と声をかける Doing は同じでも、"競争者としての Being" から出たそれと、"見守る Being" から出たそれは、受け手の中で違う出来事になる。

### 5.2 アプリを第3層に "触ってもらう" を置く理由

`book-to-app` の第3層は「デプロイ／触ってもらう／GCで3ヶ月ループ」です。技術的にはこれらは "行動（Doing）" に見えますが、設計思想としてはすべて **Being** の実装です。

- **デプロイ**：v0.1を出す = "完璧を待つ Being" から "対話を始める Being" への転換
- **触ってもらう**：明日1人に手渡す = "作者の Being" から "対話者の Being" への転換
- **GCで3ヶ月ループ**：場に返し続ける = "個人の学び" から "コミュニティの Being" への転換

これはランドマーク教育や Erhard & Jensen の系譜そのままではなく、**ありえる楽考のたまかつ** に翻訳された Being の運用です（参照：[[30_概念/たまかつ]]）。「対等・率直・肯定」の3原則が、Being の日常運用の言語になっている。

### 5.3 なぜ Being を教えるのは難しいか

Being は Doing に還元できないので、**"教える" ことが原理的に難しい**。教えられるのは Doing だけ。Being は **顕れる** ものです。

だから `book-to-app` のL4の記述は、**Being を教えられないが、Being が顕れる場を設計する** ことを狙います：

- L4（deploy）：公開直後の "誰に最初に触ってもらうか" まで設計できている
- L4（touch_together）：触った人の反応から、次のテーマ／次のアプリを設計できる
- L4（gc_loop）：参加者の変化に応じて、テーマの順序・深さを動的に組み替えられる

これらは **Doing の記述** ですが、**その Doing の背後にある Being** を暗黙に指す。**「あなたが Being を持っているとき、こういう Doing が生まれる」** の逆算になっている。

> **🌱 深めるための問い**：あなたの本の第3層に "在り方" を置くとしたら、その本の著者は何者としてその場に立っているか？

## 第6部｜自分でルーブリックを書く実務

ここから先は理論の実装編です。実際に手を動かすときの原則をまとめます。

### 6.1 3領域の見つけ方

`_rubric_コンサルタントコーチとしての熟達` は 🔍見立て（Seeing）／🎯介入設計（Intervening）／🌱在り方（Being）の3領域でした。あなたの本にも3領域を見つけましょう：

**質問1**：この本の主張が実現するには、何と何と何が揃う必要があるか？

- 「たまかつ」なら → 気づく（自覚）／立ち上がる（宣言）／生きる（実践）
- 「NVC」なら → 観察／感情／ニーズ ＋ 依頼
- 「KPI」なら → 測る／読む／動かす

**質問2**：3領域は "並列" か "順序" か？

順序なら、L1〜L4は3領域を横断して1つの旅になる。並列なら、3領域それぞれにL1〜L4がある。`_rubric_コンサルタントコーチとしての熟達` は "並列" で書かれています（3領域それぞれに4段階）。

### 6.2 4段階の書き方 ─ 現場の具体行動から書く

**最悪の書き方**：抽象な形容詞で書く。「L1は不十分、L2は基礎的、L3は良好、L4は卓越」→ 誰も動けない。

**良い書き方**：**現場の具体行動を1つ書く**。「L1は "売上が下がっている" とクライアントに言われて、そのまま処方箋を出そうとする」。

原則：
1. **L1** は "誰もが通る" 具体行動。恥じるためではなく "気づいていない状態" を可視化する
2. **L2** は "教科書に書いてある" の再現。フレームワーク・型・質問リスト
3. **L3** は "型の外" が見える具体行動。フレームが機能しない固有性を掴む
4. **L4** は "原理から立ち上げる" 具体行動。設計思想そのものを他者に教えられる

**書き方のコツ**：**L1とL4を先に書く**。次に L2 と L3 を "L1からL4への移行" として書く。L1→L4を先に決めると、旅の距離が決まる。

### 6.3 次段階への Slow Question

各段階に **"次段階へのSlow Question"** を添えると、地図に "歩き方" が乗ります。書き方の原則：

- **一段先** を狙う（二段先は届かない）
- **問いの主語は "あなたは"**（相手の位相を動かす）
- **判断を求めない**（Yes/Noで答えられる問いは深まらない）
- **その場で答えられなくてよい** 重さを持たせる（Slow で動く問いだから）

**悪い例**：「L1のあなたは、L2に進むために本を読みましたか？」（Yes/No、判断、進歩を求める）

**良い例**：「あなたはいま、クライアントの言葉を "そのまま" 受け取っていないか？ その症状の背後に、どんな型で見るとどう見えるか、一度当てはめてみよう」（自省、比較、実験）

### 6.4 ルーブリックを "書き直す" 練習

**練習1**：既存のアプリ（tamakatsu / KPI / NVC など）の1テーマを開き、そのルーブリックのL1を **あなたの現場の具体行動** で書き直してみる。

**練習2**：あなたの本の1テーマで、L1とL4を書く。L2/L3は空欄のまま、L1→L4の "距離" を眺めてみる。

**練習3**：L1→L2の Slow Question を書く。書けなかったら、**あなた自身が L1 で誤作動している経験を思い出せていない** サイン。実体験を掘り起こす。

## 第7部｜自分で観点クイズを書く実務

### 7.1 5層フィードバックの構造

`perspective_quiz` の各選択肢は5層で構成されます：

| 層 | 役割 |
|---|---|
| `stance` | 🔍 その選択を選んだあなたの "現在の視点" |
| `phase` | ⚡🐢🌊 Fast/Slow/身体 のどの位相か |
| `phase_note` | なぜこの位相か、1行 |
| `author_view` | 📖 本文の具体箇所を引用（一段先の言語） |
| `slow_question` | 🌱 一段先の問い |

**設計手順**：
1. まず問い（q）を決める。**learning_objective**（この設問で掴みたいこと）を1行で書く
2. 4つの stance を決める（Fast:Slow:身体 = 1:2:1 の位相配分）
3. 各 stance に **選択肢テキスト（text）** を書く（読者が実際に思う言葉で）
4. 各選択肢に author_view を書く（**本文の具体引用**）
5. 各選択肢に slow_question を書く（**一段先**）

### 7.2 4択の位相配分（再掲）

- ⚡ Fast の反射：1つ
- 🐢 Slow の再構築：2つ（**"自分に届く方" を選ばせる**）
- 🌊 身体の感覚：1つ

Slow が2つある意味を再度強調：**同じ主題を "設計視点" と "現場視点" のように2種類の Slow で見せる** と、読者は自分に近い方を選べる。これが Comprehensible Input の「i+1」を複数用意する実装。

### 7.3 author_view で本文を引用する原則

**必須ルール**：`author_view` は必ず本文の具体箇所を引用する。抽象要約に逃げない。

**なぜか**：
- 抽象要約は多くの場合 i+2 以上に飛ぶ
- 本文の引用は "著者が読者の現在地から書いた一段先" である確率が高い
- 引用は **読者を本に戻す動線** になる（アプリは本を代替しない）

**引用が難しい場合**：
- OCR素材が薄い → 素材の質＝アプリの質。OCRを厚くする方が先
- 抜き書きが抽象的 → 「例」「事例」「エピソード」の部分を優先して抜く

### 7.4 slow_question の重さの計り方

`slow_question` の重さは、**"読者が今日1つ動ける重さ"** に合わせます。重さの目安：

| 重さ | 例 | 使いどころ |
|---|---|---|
| 軽い | 「その本に戻るとき、あなたは何を確かめに戻っているか？」 | L1→L2 の入り口 |
| 中 | 「フレームワークが機能しない固有の文脈は何か？」 | L2→L3 |
| 重い | 「クライアントの心的表象のどの一点を書き換えると、全体が動くか？」 | L3→L4 |

**間違い**：L1 向けのクイズに、L3→L4 級の重い問いを添える。→ 届かない。

**確かめ方**：**あなた自身がその問いを今日1つ動ける** かどうかで確かめる。動けなかったら重すぎる。

## 第8部｜v0.1 → 対話 → v0.2 の改善ループ

### 8.1 v0.1 は "触ってもらう" ためにある

`book-to-app` の第3層（deploy / touch_together / gc_loop）が伝える通り、アプリは公開してから始まります。v0.1 の目的は **触ってもらって、あなたの設計の "見えていなかったもの" を発見する** こと。

### 8.2 対話から拾うべき信号

触ってもらった時、読者から拾うべきは "評価" ではなく **"位相の変化"** です。

**拾う信号**：
- どの選択肢に "手が止まった" か（Learning Zone に居た瞬間）
- どの選択肢が "自分にはない視点だった" か（i+1 が届いた瞬間）
- どの slow_question が "帰宅後も残っていた" か（Fast への沈殿の入り口）

**拾わなくていい信号**：
- 「面白かった」「勉強になった」（表層の反応、位相の変化を示さない）
- 「難しかった」（二段先だったサイン。単に軽くすればいいわけではない）

### 8.3 v0.2 で直すべきもの

**優先1**：author_view が抽象になっているところ → 本文の具体箇所に差し替える
**優先2**：slow_question が二段先になっているところ → 一段先に降ろす
**優先3**：4択の位相配分が Slow に偏りすぎ／Fast に偏りすぎているところ → 1:2:1 に近づける
**優先4**：ルーブリックのL1が抽象になっているところ → 現場の具体行動に書き換える

**後回しでいいもの**：
- CSS の見た目
- 動画URLの埋め込み（対話が回り始めてから）
- テーマ数の増減（11前後は経験的に良い数字）

### 8.4 改善が止まらないための「触ってもらう相手」の設計

改善のループが止まる時、原因はほぼいつも **"触ってもらう相手" が固定化している** ことです。

- 同じ人ばかりに触ってもらう → 同じフィードバックしか得られない
- 手渡す相手が同じ位相 → i+1 の照準が偏る

**推奨**：3ヶ月ごとに **"新しい触り手"** を最低1人加える。位相の違う読者を並行して持つと、v0.n の改善が別の角度から見えます。

## エピローグ｜あなたはこのガイドを "読んだ" のか、"生きる" のか

Ericsson の言葉を借りて閉じます。

> **「知っている（Know）」と「できる（Do）」の間には距離がある。「できる」と「Being として現場に立てる」の間にはもっと距離がある。**

このガイドをここまで読んだあなたは、`book-to-app` の設計思想を **知った** 状態です。**できる** 状態にするには、自分の本で1つルーブリックを書き直してみる。**Being として立てる** 状態にするには、それを誰かに触ってもらって、対話を回し続ける。

L4は到達点ではなく、**毎日再更新される熟達の位相**。あなたが原則を持って改造したその瞬間から、`book-to-app` は **あなたのアプリ** になります。
